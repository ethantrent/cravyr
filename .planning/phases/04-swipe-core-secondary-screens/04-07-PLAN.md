1. Add POST / handler to `apps/api/src/routes/saves.ts`:

   Insert ABOVE the existing `savesRouter.delete('/:id', ...)` handler. The handler:
   - Extracts `userId` from `req.user?.id` (JWT verified by requireAuth middleware -- NEVER from request body)
   - Reads `restaurant_id` (string, required) and `interaction_type` (string, defaults to `'right'`) from `req.body`
   - Validates `restaurant_id` is a non-empty string; returns 400 if missing
   - Validates `interaction_type` is one of `['right', 'superlike']`; returns 400 if invalid
   - Inserts into the `saves` table: `{ user_id: userId, restaurant_id, interaction_type, saved_at: new Date().toISOString() }`
   - Uses `.upsert()` with `onConflict: 'user_id,restaurant_id'` to handle re-saving (idempotent)
   - Returns 201 with the created/updated save record `{ id, interaction_type }`
   - On Supabase error, returns 500 with error message

   ```typescript
   savesRouter.post('/', async (req: Request & { user?: { id: string } }, res: Response) => {
     const userId = req.user?.id;
     if (!userId) {
       res.status(401).json({ error: 'Unauthorized' });
       return;
     }

     const { restaurant_id, interaction_type = 'right' } = req.body as {
       restaurant_id: string;
       interaction_type?: string;
     };

     if (!restaurant_id || typeof restaurant_id !== 'string') {
       res.status(400).json({ error: 'restaurant_id is required' });
       return;
     }

     const validTypes = ['right', 'superlike'];
     if (!validTypes.includes(interaction_type)) {
       res.status(400).json({ error: 'interaction_type must be right or superlike' });
       return;
     }

     const { data, error } = await supabaseAdmin
       .from('saves')
       .upsert(
         { user_id: userId, restaurant_id, interaction_type, saved_at: new Date().toISOString() },
         { onConflict: 'user_id,restaurant_id' }
       )
       .select('id, interaction_type')
       .single();

     if (error) {
       res.status(500).json({ error: error.message });
       return;
     }

     res.status(201).json(data);
   });
   ```

   SECURITY NOTE: userId comes from JWT (req.user.id), not from request body. This prevents users from creating saves on behalf of other users. The requireAuth middleware (already applied via `savesRouter.use(requireAuth)`) handles JWT verification.

2. Add DELETE /:id handler to `apps/api/src/routes/swipes.ts`:

   Insert BELOW the existing `swipesRouter.post('/', ...)` handler. The handler:
   - Extracts `userId` from `req.user?.id` (JWT verified by requireAuth middleware)
   - Reads `id` from `req.params` (this is the restaurant_id, since discover.tsx calls `DELETE /api/v1/swipes/${restaurant.id}`)
   - Deletes from `swipes` table where `restaurant_id = id AND user_id = userId` (dual filter for ownership)
   - Returns 204 on success (even if count is 0 -- idempotent delete)
   - On Supabase error, returns 500 with error message

   ```typescript
   swipesRouter.delete('/:id', async (req: Request & { user?: { id: string } }, res: Response) => {
     const userId = req.user?.id;
     if (!userId) {
       res.status(401).json({ error: 'Unauthorized' });
       return;
     }

     const { id } = req.params;

     const { error } = await supabaseAdmin
       .from('swipes')
       .delete()
       .eq('restaurant_id', id)
       .eq('user_id', userId);

     if (error) {
       res.status(500).json({ error: error.message });
       return;
     }

     res.status(204).send();
   });
   ```

   IMPORTANT: The `:id` param is the `restaurant_id` (not the swipe record UUID). This matches the client call in discover.tsx line 87: `fetch(\`${API_URL}/api/v1/swipes/${restaurant.id}\`)`. The dual filter (restaurant_id + user_id) ensures a user can only delete their own swipe record. Returns 204 regardless of whether a row was deleted (idempotent) because the undo operation is best-effort.

   No changes needed to server.ts -- both routers are already registered.

| T-04-07-05 | Denial of Service | POST /api/v1/saves upsert | mitigate | Upsert on (user_id, restaurant_id) prevents duplicate row accumulation from repeated save attempts. Rate limiting is an infrastructure concern handled at the Express middleware level (helmet/rate-limit from Phase 2). |
