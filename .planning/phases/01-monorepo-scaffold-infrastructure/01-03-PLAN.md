2. Create `.github/workflows/supabase-keepalive.yml` with the following exact content (per D-05 — GitHub Actions cron, every 3 days, SELECT 1 equivalent):

```yaml
name: Supabase Keep-Alive

on:
  schedule:
    # Run every 3 days at 9:00 AM UTC to prevent Supabase free-tier 7-day auto-pause
    - cron: '0 9 */3 * *'
  workflow_dispatch: # Manual trigger for testing

jobs:
  ping:
    runs-on: ubuntu-latest
    steps:
      - name: Ping Supabase database
        env:
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_ANON_KEY: ${{ secrets.SUPABASE_ANON_KEY }}
        run: |
          HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
            "${SUPABASE_URL}/rest/v1/?apikey=${SUPABASE_ANON_KEY}&limit=0")
          echo "Supabase responded with HTTP ${HTTP_STATUS}"
          if [ "${HTTP_STATUS}" -ne 200 ]; then
            echo "ERROR: Supabase ping failed with status ${HTTP_STATUS}"
            exit 1
          fi
          echo "Supabase keep-alive ping successful"
```

This uses the curl approach (simpler, no Node.js setup needed) from RESEARCH.md. It hits the PostgREST endpoint which counts as database activity, preventing the 7-day auto-pause.

**Repository secrets required:** The workflow needs `SUPABASE_URL` and `SUPABASE_ANON_KEY` set as GitHub repository secrets. These can be set via the GitHub CLI:
```bash
gh secret set SUPABASE_URL --body "https://dxkvtcpgqkbkhjshvqji.supabase.co"
gh secret set SUPABASE_ANON_KEY --body "<anon-key-value>"
```

Or via GitHub UI: Repository Settings > Secrets and variables > Actions > New repository secret.

Attempt to set the secrets via `gh secret set` if the GitHub CLI is authenticated. Use the Supabase URL from memory (https://dxkvtcpgqkbkhjshvqji.supabase.co) and the anon key from memory. If `gh` is not authenticated or the command fails, note it in the summary — the user can set them manually.
